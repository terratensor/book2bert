## 8.1 Что мы сделали

Мы прошли полный цикл создания BERT-модели с нуля:

1. **Подготовили данные** — 11,361 книга военно-исторического корпуса militera (14 ГБ текста, 93M предложений)
2. **Обучили токенизатор** — WordPiece с vocab_size=50,000 на militera корпусе

4. **Реализовали BERT-tiny** — 29.9M параметров, 6 слоёв, 384 hidden size
5. **Обучили модель** — 5 эпох, ~55 часов на одной GPU (RTX 5060 Ti 16 GB)
6. **Протестировали** — MLM, многотокенные слова, [CLS] эмбеддинги
7. **Сравнили с HF моделями** — rubert-tiny, distilrubert-tiny, rubert-base, bert-base-multilingual

## 8.3 Главные уроки

### 1. Качество данных важнее архитектуры

Самым важным этапом оказалась **подготовка данных**, а не выбор модели.

**Что мы сделали правильно:**
- Обучили токенизатор на militera корпусе (14 ГБ), а не на 50 книгах (27 МБ)
- Отфильтровали CJK и тайские символы
- Разбили текст по `\n` для обработки списков
- Добавили `[SEP]` между предложениями

**Что было ошибкой:**
- Первый эксперимент на 50 книгах с vocab_size=30k → модель не выучила семантику
- Отсутствие `[SEP]` между предложениями в v2 → модель хуже понимала структуру

**Вывод:** 80% успеха — в данных. 20% — в архитектуре.

### 2. Токенизатор на большом корпусе — ключевой фактор

| Корпус для токенизатора | Результат |
|------------------------|-----------|
| 50 книг (27 МБ) | Субворды бессмысленные (`невзгод` → `['нев', '##з', '##го', '##д']`) |
| militera (14 ГБ) | Субворды осмысленные (`невзгод` → `['невз', '##год']`) |

**Вывод:** Токенизатор нужно обучать на максимально большом и разнообразном корпусе. Даже если сама модель будет обучаться на подмножестве.

### 3. Специализация побеждает размер

Наша tiny-модель (30M) на военных текстах оказалась лучше:
- distilrubert-tiny (107M) — на 10%
- rubert-tiny (30M) — в 3.8 раза

**Вывод:** Универсальная модель, обученная на Википедии и новостях, не понимает военную терминологию. Специализированная модель на нужном корпусе даёт лучшее качество при меньшем размере.

### 4. Python + Go — синергия для больших данных

| Задача | Инструмент | Время |
|--------|-----------|-------|
| Обработка корпуса (11k файлов) | Go (горутины) | 2-3 часа |
| Сбор корпуса для токенизатора | Go | 54 секунды |
| Сегментация предложений | Python (HTTP сервис) | 2-3 часа |
| Обучение токенизатора | Python | 2 минуты |
| Сборка датасета | Python (с батчевой токенизацией) | 17 минут |
| Обучение модели | Python (PyTorch) | 55 часов |

**Вывод:** Go хорош для IO-связанных задач (чтение файлов, HTTP). Python — для ML-вычислений. Их комбинация даёт оптимальный результат.

### 5. Итеративная отладка — единственный путь

Мы прошли через несколько неудачных экспериментов:

| Эксперимент | Проблема | Решение |
|-------------|----------|---------|
| 50 книг, vocab=30k | Модель не выучила семантику | Увеличили корпус до 14 ГБ |
| v2 (без `[SEP]`) | Модель не видела границы предложений | Добавили `[SEP]` между предложениями |
| Медленная сборка датасета (6-8 часов) | Построчная токенизация | Батчевая токенизация (`encode_batch`) |
| Двойной `[CLS]` и `[SEP]` | Ручное добавление | Убрали, оставили автоматическое |

**Вывод:** Не бойтесь ошибаться. Каждая ошибка — это урок, который делает финальный результат лучше.

## 8.4 Что мы сделали неправильно (и как исправили)

| Ошибка | Последствия | Исправление |
|--------|-------------|-------------|
| Обучили токенизатор на 50 книгах | Субворды бессмысленные | Переобучили на militera (14 ГБ) |
| Не добавили `[SEP]` между предложениями | Модель не видела границы | Добавили `[SEP]` при склейке |
| Двойной `[CLS]` и `[SEP]` | Лишние токены | Убрали ручное добавление |
| Построчная токенизация | Сборка датасета 6-8 часов | Батчевая токенизация → 17 минут |
| Не фильтровали CJK | Мусор в словаре | Фильтрация на всех этапах |
| Не разбивали по `\n` | Списки склеены в одно предложение | Разбивка в `process-corpus` |

## 8.5 Что можно улучшить

### Ближайшие шаги (по приоритету)

| Приоритет | Действие | Ожидаемый эффект | Затраты |
|-----------|----------|------------------|---------|
| 1 | Увеличить эпохи до 20-30 | Val loss: 2.58 → 2.4-2.5 | ~200 часов |

### Что НЕ нужно делать

| Идея | Почему нет |
|------|------------|
| Увеличить batch size | Нет памяти (16 GB) |
| Перейти на BERT-base без увеличения данных | Бесполезно (эксперимент с 50 книгами доказал) |
| Добавить NSP | Для semantic search не нужно |
