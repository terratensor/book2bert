# План создания идеального датасета для BERT

## Цель
Из текущего корпуса (1 млрд предложений, 450 GB, 176k JSONL файлов) получить **чистый, сбалансированный, дедуплицированный корпус** для обучения BERT-small и последующих моделей.

## Исходные данные
| Параметр | Значение |
|----------|----------|
| Расположение | `/mnt/archive/book2bert/data/processed/sentences_full` |
| Формат | JSONL (каждая строка — предложение с метаданными) |
| Количество файлов | 176,407 |
| Количество предложений | ~1,081,394,263 |
| Размер на диске | 408 GB |

**Решение:** Используем текущий корпус как есть. Анализ покажет, нужны ли доработки.

## Фаза 2: Грубая фильтрация (один проход, Go)

### 2.1 Критерии удаления предложения
| Критерий | Порог (будет уточнён после Фазы 1) | Пример |
|----------|-----------------------------------|--------|
| Пустая строка | длина == 0 | "" |
| Слишком короткое | < N символов (N = 20-50) | "Война" |
| Слишком длинное | > M символов (M = 1000-2000) | (юридические тексты) |
| Слишком много цифр | > P% цифр (P = 20-40) | "8.29 4.6 5.0 7,4 9.5..." |
| Слишком много пунктуации | > Q% пунктуации (Q = 40-50) | "---!!!???..." |
| Начинается с маркера списка | "•", "-", "1.", "1)", "a)" | Списки |
| Содержит явный мусор | "ISBN", "УДК", "ББК", "www.", "http", "@" | Метаданные |
| Не русский и не английский | lang not in ['ru', 'en'] (по FastText) | Другие языки |
| Слишком много заглавных букв | > R% заглавных (R = 50-80) | АКРОНИМЫ, КРИК |

### 2.2 Сохранение результата
- Выходной формат: JSONL (такой же, как исходный)
- Сохраняем в `sentences_filtered/`
- Сохраняем лог удалённых: `filtered_stats.csv`

## Фаза 3: Точная дедупликация (один проход, Go)

### 3.1 Алгоритм
```
хэш-таблица (в памяти или на диске)
для каждого предложения:
    h = md5(текст)
    если h в таблице:
        пропускаем (дубликат)
        увеличиваем счётчик дубликатов
    иначе:
        добавляем h в таблицу
        сохраняем предложение в sentences_dedup/
```

### 3.2 Оптимизация памяти
| Если дубликатов | Решение |
|----------------|---------|
| Мало (<10%) | Хранить все хэши в памяти (максимум 16 GB) |
| Много (>10%) | Использовать Bloom filter + дисковая проверка |

### 3.3 Сохранение результата
- `sentences_dedup/`
- Лог: `dedup_stats.csv`

## Фаза 4: Семантическая дедупликация (опционально, Python)

### 4.1 Когда нужна
- Если после точной дедупликации осталось >200 GB
- Если есть подозрение на семантические дубликаты (разные формулировки одного смысла)
- **Решение:** после Фазы 3 оцениваем объём и качество

### 4.2 Алгоритм (SemHash)
```
для каждого предложения:
    эмбеддинг = sentence_transformer.encode(текст)
    хэш = семантический_хэш(эмбеддинг)
    если хэш в таблице:
        проверяем косинусное сходство с существующим
        если >0.85:
            пропускаем
        иначе:
            сохраняем
    иначе:
        сохраняем
```

### 4.3 Стоимость
- Время: 2-4 дня на 100 GB
- Память: 32 GB для хранения эмбеддингов

## Фаза 5: Балансировка корпуса (один проход, Python)

### 5.1 Цели балансировки
| Аспект | Цель | Почему |
|--------|------|--------|
| Длина предложения | Нормальное распределение | Без перекоса в короткие/длинные |
| Язык | Пропорционально реальному (или каппинг 30%) | Модель не должна доминировать один язык |
| Жанр (если есть метаданные) | Равномерно | Обобщение, а не переобучение |

### 5.2 Стратегия сэмплирования
```python
# Взвешенная выборка
вес = 1 / (частота_языка * частота_длины)

# Или каппинг (ограничение максимальной доли)
if доля_языка > 0.3:
    пропускаем_с_вероятностью = (доля_языка - 0.3) / доля_языка
```

### 5.3 Сохранение
- `sentences_balanced/`
- Лог: `balancing_stats.csv`

## Фаза 7: Решение об объёме обучения

### 7.1 Оценка финального датасета
| Размер | Решение |
|--------|---------|
| <50 GB | Берём всё |
| 50-200 GB | Берём всё, если позволяет время |
| >200 GB | Берём случайную выборку 20-30% (уже чистую) |

### 7.2 Случайная выборка из чистого корпуса
- Если нужно уменьшить объём, берём выборку **после** всей очистки
- Используем reservoir sampling на Go
